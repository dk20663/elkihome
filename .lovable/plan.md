# Keyword-шаги: сканирование всей цепочки

## Новое поведение

Когда наступает время обработать чат:

1. Берём текущий шаг по `current_step`.
2. **Если шаг без ключевых слов** — работаем как сейчас (отправляем по расписанию, переходим к следующему).
3. **Если шаг с ключевыми словами** — запускаем «keyword-скан»:
   - Собираем подряд **все** идущие keyword-шаги, начиная с текущего, до первого обычного шага (или конца цепочки).
   - Из них отфильтровываем те, что уже были отправлены в этом чате (по `avito_message_log`).
   - Среди оставшихся ищем шаг, у которого хотя бы одно ключевое слово найдено в последних 3 сообщениях клиента.
   - **Если нашёлся** — отправляем именно его. `current_step` сдвигаем сразу за этот шаг. Промежуточные keyword-шаги остаются «не отправленными» и при следующем входящем сообщении снова попадут в скан.
   - **Если не нашёлся ни один** — ничего не отправляем, `current_step` не двигаем. Чат ждёт следующего сообщения клиента (webhook обновит `next_run_at`).
4. После отправки — если следующий шаг (уже не keyword) имеет `delay = 0`, продолжаем внутренний цикл, как сейчас.

## Что это меняет на вашем примере

Шаги 1, 2 отправлены. Шаг 3 — keyword «A». Шаг 4 — keyword «B». Клиент пишет слово «B».

- Сканируем оба keyword-шага (3 и 4).
- Шаг 3 не подходит, шаг 4 подходит → отправляем шаг 4.
- `current_step` становится 5.
- Если позже клиент напишет слово «A» — шаг 3 уже **пропущен навсегда** (он позади курсора), реакции не будет.

Если нужна реакция на «A» даже после ухода вперёд — это уже другая модель (отдельный пул реакций), её обсудим отдельно при необходимости.

## Идемпотентность

Сохраняется существующая защита: перед отправкой проверяем `avito_message_log` (`chat_id + step_id + status='sent'`). Один и тот же шаг не уйдёт дважды даже при retrigger цепочки.

## Граничные случаи

- **Нет keyword-шагов подряд** (только один) — поведение совпадает с текущим.
- **Все keyword-шаги в группе уже отправлены** — пропускаем всю группу, переходим к следующему обычному шагу.
- **Несколько шагов в группе совпали по словам** — отправляем первый по `order_index` (детерминированно).
- **Обычный шаг между keyword-шагами** — он разделяет группы; скан не «прыгает» через обычные шаги.

## Технические детали

Файл: `supabase/functions/avito-process-chains/index.ts`

В существующем `while (true)` блоке заменить ветку `if (step.keyword_triggers && length > 0)`:

```text
collect group = [step, steps[i+1], ...] пока keyword_triggers.length > 0
загрузить из avito_message_log все step_id со status='sent' для chat_id ∩ group
отфильтровать group → candidates (не отправленные)
если candidates пуст:
    current_step = индекс_за_группой; next_run_at = null; break
найти первый candidate, у которого keyword совпал с clientHaystack
если найден:
    проверить идемпотентность (как сейчас)
    отправить, current_step = его_индекс + 1, продолжить внешний while
если не найден:
    next_run_at = null; break  (ждём следующего сообщения клиента)
```

Логику обычных (без keywords) шагов и delay=0-цепочки не трогаем.

Изменений в БД и UI не требуется.
